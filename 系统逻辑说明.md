# 活动提醒系统逻辑说明

## 数据来源

核心数据来自两个文件：

- `102国服活动排期表.xlsx`
- `活动与礼包对应关系.xlsx`

`daily_reminder.py` 读取 `102国服活动排期表.xlsx` 生成提醒数据。

`活动与礼包对应关系.xlsx` 用来把活动名映射成活动 ID 或展示名称，例如：

```text
29145: 火力全开:进攻土拨鼠128~648
1000125:VIP商店
```

## 设置日期规则

每个活动先从排期表读到“开启日期”，再通过 `reminder_rules()` 计算“设置日期”。

规则如下：

```text
区域迁徙：
  开始周的前一周周一：设置礼包+分组
  再过 2 天：设置活动+公布分组

本服联盟GVE：
  开始周的前一周周一设置

跨服联盟GVE / 通服跨服联盟GVE：
  开始周前 2 天设置

VIP商店 / 黑骑士：
  开始周周一设置

联盟远征 / 功勋商店：
  开启当天设置

土拨鼠：
  开启前 2 天设置

自选周卡 / 进化连冲：
  开启前 3 天设置

其他活动：
  开启前 1 天设置
```

## 活动名称识别

`activity_name()` 按关键词识别活动类型。

例如：

```text
土拨鼠、罩子、邮件
```

会被特殊识别为：

```text
土拨鼠、罩子、邮件
```

普通土拨鼠会通过映射表显示为：

```text
29145: 火力全开:进攻土拨鼠128~648
```

## 时间生成规则

默认开始时间：

```text
开启日期 08:00:00
```

结束时间：

```text
开始时间 + 活动天数 - 1 秒
```

例如 1 天活动：

```text
2026-06-21 08:00:00 ~ 2026-06-22 07:59:59
```

特殊活动：

```text
特化蚁对决 / 蚁后对决
```

开始时间为：

```text
08:15:00
```

## 服务器识别规则

服务器优先通过 `computed_server_text()` 自动推算。

部分规则如下：

```text
VIP商店：
  以 2026-06-20 为基准，每周服务器 +1

土拨鼠：
  以 2026-06-14 为基准，每 14 天服务器 +2

联盟远征：
  以 2026-06-15 为基准，每周服务器 +1

本服联盟GVE：
  以 2026-06-22 为基准，每周服务器 +1

跨服联盟GVE：
  以 2026-06-15 为基准，每周服务器 +1
```

如果不能自动推算，会从原始内容里识别：

```text
S1-S854
1~854
Sxxx
```

如果仍然无法识别，则显示：

```text
排期表未标明
```

## 土拨鼠、罩子、邮件特殊逻辑

只要原始内容同时包含：

```text
土拨鼠
罩子
邮件
```

就不再按普通土拨鼠展示，而是固定展示为三块。

模板如下：

```text
土拨鼠服：
29145: 火力全开：进攻土拨鼠128~648
时间：开启日 08:00:00~次日 07:59:59 UTC+8
新服不自动开启，每日刷新 pop2

区域征战服：
保护罩【开启日 08:00:00~次日 07:59:59 UTC+8】定时24h【赛季外，区域征战范围服 】

邮件，赛季外：
定时：开启日前一天 16:00:11
204115：野任饲料*30000
202042: 2小时兵蚁解化加速*10
202044:2小时治 加速*10
204042：奇异密雯V*1
200077: 100钻石*4
203001：巢六保护8小时*1
```

例如活动开启日期为 `2026-06-21`，则邮件定时为：

```text
2026-06-20 16:00:11
```

## 生成文件

运行：

```bash
python3 daily_reminder.py
python3 build_reminders_html.py
```

会生成：

```text
每日活动提醒.txt
全部每日提醒.html
index.html
site/index.html
```

`index.html` 是 GitHub Pages 使用的根目录页面。

## HTML 页面逻辑

`build_reminders_html.py` 会生成完整网页。

页面逻辑：

```text
只展示设置日期 >= 当前北京时间 的提醒
按设置日期分组
每组下面展示活动卡片
顶部显示“今天要设置什么”和“下一次要设置什么”
支持搜索
支持全部展开/全部收起
```

页面前端也会按北京时间动态隐藏已经过去的日期。

```text

```

## 本地一键同步逻辑

`一键同步GitHub.command` 用于 Mac。

它会执行：

```text
1. 进入项目目录
2. 找 python3 / python
3. 运行 daily_reminder.py
4. 运行 build_reminders_html.py
5. git add .
6. 如果有变化就 git commit
7. git pull --rebase origin main
8. git push origin main
```

如果显示：

```text
没有检测到需要提交的修改
```

说明文件内容和上一次提交一致。

如果显示：

```text
ahead 1
```

说明本地有提交还没有推到 GitHub。

## GitHub 自动更新逻辑

`.github/workflows/update-pages.yml` 已配置自动更新。

触发时间：

```text
北京时间 00:05 自动跑一次
北京时间 08:05 自动跑一次
push 到 main 后也会自动跑
```

Actions 会运行：

```bash
python daily_reminder.py
python build_reminders_html.py
```

然后自动提交：

```text
index.html
site/index.html
全部每日提醒.html
每日活动提醒.txt
```

如果只是日期到了明天，理论上不用点 `一键同步GitHub.command`，GitHub 会自动更新。

如果修改了 `102国服活动排期表.xlsx`，需要运行：

```text
一键同步GitHub.command
```

把本地修改推送到 GitHub。

## 隐私限制

当前密码页是静态前端门禁，可以挡住普通访问，但不是严格安全的服务器级鉴权。

因为 GitHub Pages 是公开静态网页，懂技术的人仍然可以查看源码或仓库内容。

如果内容必须严格保密，建议改成：

```text
私有仓库 + 不公开 Pages
或使用带登录鉴权的托管服务
```
