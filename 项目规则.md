# 102国服活动排期表 — 项目规则

## 一、项目用途

从活动排期表自动生成每日活动设置提醒（TSV / TXT / HTML），并通过 GitHub Pages 发布在线页面。
仓库：`github.com/leeyeongjun1234-hash/102guofu-activity-reminders`（Pages: main 分支根目录）。

## 二、文件角色

| 文件 | 角色 | 可否手改 |
|---|---|---|
| `102国服活动排期表.xlsx` | 核心数据源。**实际是 TSV 文本文件**，仅扩展名为 .xlsx | ✅ 手动维护 |
| `工作休日日历.xlsx` | 真 Excel。`每日明细` 表（日期/状态/是否休息）决定工作日 | ✅ 手动维护 |
| `活动与礼包对应关系.xlsx` | 真 Excel。活动与礼包 ID 映射 | ✅ 手动维护 |
| `活动设置提醒.tsv`、`每日活动提醒.txt`、`全部每日提醒.html`、`index.html`、`site/index.html` | 生成产物 | ❌ 勿手改，会被覆盖 |
| `generate_reminders.py` → `daily_reminder.py` → `build_reminders_html.py` | 生成脚本，须按此顺序执行 | 改逻辑时改这里 |
| `workday_calendar.py` | 工作日/固定周末规则的公共模块 | 同上 |
| `自动更新HTML.command` | 本地监听（3 秒轮询），源文件一变即重建 | — |
| `一键同步GitHub.command` | pull --rebase → 重建 → commit → push | — |

## 三、数据源格式规则（重要）

1. `102国服活动排期表.xlsx` 必须保持 **UTF-8 制表符分隔文本**。不要用 Excel 另存为真正的 .xlsx（zip 格式），否则所有脚本读取失败。
2. 第 2 行是日期行，格式 `M月D日`；年份在脚本中写死为 **2026**，跨年需改 `YEAR` 常量。
3. 每行前 4 列作为该行的上下文（用于识别活动 ID）；日期列填活动内容，空即无活动。
4. `工作休日日历.xlsx` 的 `每日明细` 表必须有 `日期` 列；`是否休息`（1/是/休/true）或 `状态`（含"休"且不含"上班"）判定休息日。日历中没有的日期默认周一至周五为工作日。

## 四、提醒时间规则（设置日 = 相对活动开启日）

| 活动类型 | 设置时间 |
|---|---|
| 区域迁徙 | 提前一周的周一：设置礼包+分组；周三：设置活动+公布分组 |
| 本服联盟GVE | 提前一周的周一 |
| 跨服联盟GVE | 开启周前的周六（周一前 2 天） |
| VIP商店、黑骑士 | 开启当周周一 |
| 联盟远征、功勋商店 | 开启当日 |
| 火力全开/土拨鼠礼包（29145/32364） | 提前 1 天 |
| 土拨鼠（其他） | 提前 2 天 |
| 自选周卡、进化连冲 | 提前 3 天 |
| 其他所有活动 | 提前 1 天 |

**休息日调整**：设置日落在休息日时，自动提前到最近的工作日。
**固定例外（不调整）**：`1000905 国服-拯救蚜虫-跨服`、`1000927 飞蜥之战` 保持休息日/周末设置；飞蜥之战固定开启周前的周日设置。

## 五、内容规范化规则

- **通服跨服联盟GVE 服务器识别**：`拯救蚜虫-跨服` 这类单元格中「通服跨服联盟GVE S859 … S1-S857」的写法，服务器范围识别为「S1-S857，S859」（常规范围 + 单独的通服服务器）。逻辑在 `daily_reminder.py` 的 `server_text()`。

- 土拨鼠礼包行统一替换为 `32364: 火力全开：进攻土拨鼠（26/6/30版本）`，保留区服范围后缀。
- `进化连冲` 展示时展开为 4 条充值活动（1900133/1900162/1900163/1900084）。
- 落潮海岸、黑骑士、珍宝商店、自选周卡、特化蚁对决、蚁后对决等强制带活动 ID 展示；特化蚁对决、蚁后对决提醒 8:15 设置。
- 各活动持续天数以 `daily_reminder.py` 的 `DURATIONS` 表为准（如黑骑士 1 天、珍宝商店 6 天、蚁后对决 45 天）。
- **配套礼包通用识别**：排期表单元格内、活动标题下方以空行分隔的段落，凡含 5~6 位 ID 的礼包行（如 `29583: xxx`），自动识别为"配套礼包"分组，随卡片和每日提醒一起展示；紧跟礼包前的"服务器："行视为分组归属（如荣耀庆典按服务器段分组），段内的服务器/时间/每日刷新/新服行作为该组附加信息。少数活动（蜂群宝藏、惊喜转盘、无限寻宝、双倍盛典、落潮海岸、蚁群派对、蚁群狂欢、四个主题活动）仍走 `build_reminders_html.py` 里的专属展示逻辑。逻辑位于 `daily_reminder.py` 的 `generic_package_blocks()`。

## 六、日常操作流程

1. **改表前**先从 GitHub 拉取最新版本（或直接跑一键同步脚本，它会先 pull）。
2. 本地预览：双击 `自动更新HTML.command`，保持窗口开着，保存任一源表即自动重建。
3. 发布上线：双击 `一键同步GitHub.command`。
4. 同一时间只允许一人编辑排期表，避免冲突；同步冲突时不要强推，先人工确认哪份是最新。
5. 生成产物随源文件一起提交（Actions 也会自动提交），不要加入 .gitignore。

## 七、自动化

- GitHub Actions（`update-pages.yml`）：每天北京时间 00:05 和 08:05 定时重建；三个源表或任一脚本 push 到 main 也会触发；自动提交生成文件（commit 信息 `chore: update reminder page`）。
- 修改脚本或工作流后，记得同时 push，线上定时任务才会用到新逻辑。

## 八、修改脚本时的注意点

- 三个生成脚本共享 `week_monday`/`clean_text` 等逻辑，新增活动类型规则需同时改 `generate_reminders.py` 和 `daily_reminder.py`。
- 工作日/固定例外逻辑只改 `workday_calendar.py`（`FIXED_REST_DAY_SETUP_IDS`、`FIXED_SUNDAY_SETUP_IDS`）。
- 新活动如需固定持续天数，在 `daily_reminder.py` 的 `DURATIONS` 中登记。
